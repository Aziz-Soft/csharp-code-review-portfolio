## Issues Identified & Fixed

| # | Category | Issue | Fix Applied |
|---|----------|-------|-------------|
| 1 | 🔴 Security | Credentials hardcoded in source code | `IConfiguration.GetConnectionString()` |
| 2 | 🔴 Security | SQL Injection in `GetUserWithProfile` | Parameterized `@UserId` |
| 3 | 🔴 Security | SQL Injection in `UpdateUserEmail` | Parameterized `@Email` |
| 4 | 🔴 Security | `SELECT *` exposes `PasswordHash` | `SELECT Id, Name, Email` only |
| 5 | 🟠 Performance | `new HttpClient()` per call — socket exhaustion | `IHttpClientFactory.CreateClient()` |
| 6 | 🟠 Performance | Shared `SqlConnection` — not thread-safe | Local `using var connection` per method |
| 7 | 🟡 Logic | `reader.Read()` synchronous in async method | `await reader.ReadAsync(ct)` |
| 8 | 🟡 Logic | `NullReferenceException` if no row found | `if (user is null)` guard + `return null` |
| 9 | 🔵 Style | No `Async` suffix, sync calls, `Newtonsoft.Json` | `Async` suffix, `ExecuteNonQueryAsync`, `System.Text.Json` |

## Possible Improvements
- Wrap return in a `Result<User>` pattern to avoid exception-driven flow.
- Add `FluentValidation` at the API layer for declarative input validation.
- Replace raw ADO.NET with `Dapper` or `EF Core` for safer, cleaner queries.
